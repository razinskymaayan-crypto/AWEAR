# Design Review — Stories (bar + full-screen viewer)

**Reviewer:** Gabbana (adversarial QA lane) · **Date:** 2026-07-05
**Target:** Stories feature shipped in Phase 3 wave (shira, commit 0fe5377, merged 2026-06-30 by Jeff with syntax + no-JS-error checks only — no design gate).
**Surfaces:** `.stories-bar` in Feed (`static/index.html:523-563`, `renderStories()` ~10584) and `.story-viewer` (`:565-626`, `openStoryViewer()` ~10647). Backend `/api/stories` (`app.py:3697+`).
**Bar:** Instagram Stories / WhatsApp Status.

## Score: 4/10 — NOT demo-safe. Do not include "post a story" in the live demo script until P0s are fixed.

The static seed state (what screenshots and check-render exercise) looks fine — that is exactly the trap. The feature only holds up while `/api/stories` returns nothing. The moment one real story is posted — which is the natural "share today's outfit" demo beat — the feature collapses into raw infrastructure leaking into the UI. Verified in a real headless-Chromium render, not by code reading alone.

---

## P0 — demo blockers

### P0-1 · DATA: real stories render the client's raw IP address as the username
`_groupStories()` (index.html:10551-10559) maps `user_key` → `name`, `initials`, and the viewer header. Backend `user_key` is BE-006 = `request.client.host` or `"anon"` (app.py:3713). There is no identity join anywhere.
**Verified render:** stories bar ring labeled **"84.229.11.7"**, second ring labeled **"anon"**; viewer header shows avatar circle **"8"** (initials of an IP) + name "84.229.11.7".
**Repro:** POST one story via the Add ring (or curl), reload feed.
**Fix direction:** stories need a display identity — either join `user_key` → profile (name/avatar) server-side, or store `display_name`/`avatar` on the story row at POST time (client sends its profile), or at minimum a client-side map with a graceful "Styler" fallback. Never render `user_key` verbatim. Needs backend coordination (sam/oren) — this is not CSS-fixable.

### P0-2 · DATA: the first real story wipes Tamar/Carmel/Maayan from the bar
`loadStories()` (index.html:10575): `storyBatches = items.length ? _groupStories(items) : _seedBatches()` — real items REPLACE the seed, no merge. One posted story → the three real-user story rings vanish and the bar becomes `[Your story] [84.229.11.7]`.
**Demo consequence:** founder taps "Your story" mid-demo → the alive-looking social row self-destructs on stage.
**Fix direction:** merge seed batches with real batches (real first is fine), or seed the demo DB with stories attributed to the real users. Pair with P0-1 — fixing one without the other still fails.

### P0-3 · CONTRAST: viewer text is near-black-on-black in light theme
`.story-viewer-name`, `.story-viewer-cap`, close/del icons, and the progress-bar fill all use `var(--text)` (index.html:587-618), but the viewer surface is effectively ALWAYS dark: `.story-viewer-img` letterboxes on `background:#000` (:576). App default theme is light → `--text` is near-black.
**Verified render:** username "84.229.11.7" and caption "Today's look" are dark-gray-on-black, barely legible; the dark `text-shadow: rgba(0,0,0,.6)` (designed for light text) makes it muddier. Progress fill `var(--text)` = dark bar in a `rgba(255,255,255,.32)` track — inverted affordance.
**Fix direction:** the story viewer must be theme-independent immersive dark (Instagram/TikTok convention): inside `.story-viewer`, hard-set light-on-dark (`#fff`-family text via a scoped token, e.g. `var(--on-media,#fff)`), progress fill white, keep the dark scrims. Pure CSS, scoped to `.story-viewer *`.

## P1

1. **DATA/TIME:** `created_at = datetime.utcnow().isoformat()` (app.py:3721) is naive — no `Z`/offset. `new Date(ts)` in the browser parses it as LOCAL time, so in Israel (UTC+3) `fmtTime()` (index.html:10657-10658) computes a negative diff → every fresh story shows **"1m" for its first 3 hours**, and all times are off by the TZ offset. Fix: emit `datetime.now(datetime.timezone.utc).isoformat()` (includes `+00:00`) — additive, no schema change; frontend untouched.
2. **Ring hierarchy inverted vs the bar's own language:** unseen ring = `var(--text)` (plain black in light theme), seen = `var(--line)` — a subtle black-vs-light-gray difference that fails the 1-second scan. Meanwhile the ONLY accent-gradient ring is "Your story" (add). Instagram's grammar: gradient = unwatched content, quiet = watched, add = neutral+plus badge. Recommend: unseen = accent→accent2 gradient, seen = `var(--line)`, add ring = quiet with a small accent plus badge.
3. **A11y:** `.story-item` is click-only — no `role="button"`, no `tabindex`, no keydown. (Viewer buttons have aria-labels and 44px targets — good; tap zones are unlabeled but viewer has Esc/arrows.)
4. **DS-004:** `.story-ring{background:var(--text)}` (:540), `.story-inner{background/border var(--bg2)}` (:548-549) — tokens without fallbacks. Iron Rule as written.
5. **Seed stories have `created_at:null`** → viewer header renders name with an empty time slot; reads broken next to the Instagram norm where time is always present. Give seeds plausible relative timestamps (e.g. now-2h/now-5h/now-9h).

## P2

- No hold-to-pause; center ~30% of the viewer is a dead zone (tap zones are 35%+35%). Instagram: press-and-hold pauses, center tap advances.
- Every posted story gets the hardcoded caption `"Today's look"` (index.html:10634) — three stories in a row with identical captions read as fake. Empty caption or a one-line input sheet.
- Ring thumbnail inconsistency: seed rings show the user's avatar; real rings show the story photo with initials flashing behind while it loads. Pick one (avatar in ring is the Instagram norm; story-photo is the WhatsApp norm).
- `.story-viewer-close/.story-viewer-del { margin-left:auto }` — physical property, breaks in RTL (use `margin-inline-start`). Same debt class shira already fixed in DM bubbles.
- Bar ordering: no "my story first," no unseen-before-seen. Backend returns newest-user-first only.

## What's genuinely good
- Bar rhythm, 56px rings, `:active` scale, ellipsized 60px names — clean and native-feeling.
- Viewer architecture: progress bars with `done/active` states, 5s auto-advance, tap prev/next, Esc/arrow keys, swipe-down dismiss, owner-only delete, safe-area insets. The skeleton is Instagram-correct; it's the data layer and theming that betray it.
- Graceful seed fallback concept (bar never empty) — right instinct, wrong replace semantics.

## Verdict
Ship-blocking on data correctness, not on craft. Fix order: P0-1 + P0-2 together (identity + merge), P0-3 (scoped dark theming, ~20 lines CSS), P1-1 (one-line backend). Re-audit after fix; target 8+.
