## 1. Current state — what actually exists today

### 1.1 Hebrew string inventory (real count, not estimate)

Method: grepped `static/index.html` for the Hebrew Unicode block (`[֐-׿]`,
i.e. `[֐-׿]`).

- **552 lines** contain at least one Hebrew character (this is the line-count;
  total individual Hebrew-character occurrences are much higher since most
  lines are dense template-string HTML).
- **8,429 Hebrew characters** live inside the 70 JS render/handler functions
  (template-string-built UI). This is where the bulk of the work is — almost
  all real screen content is generated by JS, not static `<section>` HTML.
- **418 Hebrew characters** sit in static markup before the first `<script>`
  function (nav bar labels, onboarding skeleton, edit-profile modal,
  placeholders, search/explore static labels — see 1.3).
- The original ask cited "~614 hardcoded strings" — close enough to the
  552-line figure; the discrepancy is because some lines contain 2+ separate
  user-facing strings (e.g. a toast call plus a label in the same line).

**Structural i18n issue beyond strings:** `<html lang="he" dir="rtl">` is
hardcoded on line 2. This is not a string — it's a document-level attribute —
but it means even after every string is translated, the layout will render
RTL for English/other LTR users unless this is also made locale-driven.

### 1.4 Existing i18n infrastructure — found, but NOT wired up

**There is already a partial scaffold:** `static/i18n/en.json` and
`static/i18n/he.json` exist, added in the most recent commit
(`bc3504e Align Day output: tokens + reactions + i18n + ops model`,
2026-06-17). I verified:

- Both files are well-formed, parallel-keyed JSON (`app`, `nav`, `home`,
  `closet`, `feed`, `reactions`, `scan`, `chat`, `marketplace`, `wishlist`,
  `onboarding`, `rewards`, `common` — ~13 namespaces, roughly 70 leaf keys).
  `en.json` and `he.json` have matching key structure (good — whoever wrote
  this intended parity).
- **Zero references anywhere in the codebase.** `grep -rn "i18n"
  static/index.html app.py` returns nothing. No `fetch('/static/i18n/...')`,
  no `<script src=...i18n...>`, no `t(` helper function exists. The files
  are inert JSON sitting in the repo, not loaded by anything.
- Coverage is shallow relative to need: ~70 keys vs. 552 hardcoded lines —
  this scaffold covers maybe 10-12% of actual surface area (mostly nav
  labels, a few home/closet/marketplace strings, common verbs). Toasts,
  chat bot copy, sell-flow microcopy, compare/outfit-generator copy, and
  the entire admin/stylist/seasonal/sustainability surface are not
  represented at all.
- **Decision: reuse and extend this scaffold.** Someone already made the
  namespace/key-naming decision (lowercase snake_case keys grouped by
  screen) — that's consistent with the screen groupings in 1.3 above, so I
  will not propose a competing structure. I will extend `en.json`/`he.json`
  with the missing ~90% of keys rather than starting a new file format.
- No other i18n library (no i18next, no FormatJS/react-intl, no Polyglot)
  appears anywhere in `requirements.txt`, `package.json` (none exists —
  there's no Node toolchain in this repo), or `app.py`. This is a from-scratch
  vanilla-JS app with no build step, so a library with a build/transpile
  step is a non-starter anyway.

## 2. Proposed mechanism

Given: no framework, no build step, no bundler, single HTML file served
directly by FastAPI (`app.py`) as a static file.

### 2.1 Data: extend the existing `static/i18n/{locale}.json` files

Keep the namespace structure already established. Add missing namespaces as
needed (`analytics`, `sustainability`, `seasonal`, `stylists`,
`publicclosets`, `toasts`, `compare`, `outfitgen`, `explore`,
`editprofile`). Do not restructure existing keys — only add.

### 2.2 Runtime: a single small `t(key, vars?)` helper, loaded before first render

Add near the top of the existing `<script>` block (no new `<script src>`
file needed — keeps the "single file" property, though the JSON itself is
already a separate static asset and that's fine since it's data, not code):

```js
let STRINGS = {};
let LOCALE = (localStorage.getItem('awear_locale') || navigator.language || 'he').slice(0,2);
if (!['he','en'].includes(LOCALE)) LOCALE = 'en'; // fallback for unsupported locales

async function loadStrings() {
  const res = await fetch(`/static/i18n/${LOCALE}.json`);
  STRINGS = await res.json();
}

function t(key, vars) {
  const val = key.split('.').reduce((o,k) => o?.[k], STRINGS);
  let s = (val != null) ? val : key; // fallback: show the key itself, loud and visible, never blank
  if (vars) Object.entries(vars).forEach(([k,v]) => s = s.replace(`{${k}}`, v));
  return s;
}
```

`loadStrings()` must be awaited before the first `render*()` call in the
app's bootstrap sequence (wherever `showView('home')` or equivalent is
first invoked on page load) — this is the same kind of ordering hazard
flagged in the 2026-06-17 postmortem (TDZ/load-order bug class), so it
needs explicit attention, not an assumption that "it'll just work."

Document-level RTL/LTR (`<html lang dir>`) must also become dynamic:

```js
document.documentElement.lang = LOCALE;
document.documentElement.dir = (LOCALE === 'he') ? 'rtl' : 'ltr';
```

This is a structural fix, separate from string replacement, and should be
done once, early (Phase 0 below), not per-screen.

### 2.3 Conversion pattern per call site

- Static HTML: `<button data-view="home">בית</button>` →
  `<button data-view="home">${t('nav.home')}</button>` is not valid in raw
  HTML (no template literal there), so static markup needs either
  (a) an `data-i18n="nav.home"` attribute + a tiny post-load pass that
  walks the DOM and replaces `textContent` for every `[data-i18n]` element,
  or (b) converting that static HTML into a template string rendered by JS
  at boot. **Recommendation: (a)** — smaller diff, lower risk, doesn't
  disturb existing static structure/CSS selectors.
- JS template strings: `` `<div>${esc(it.name)} פריטים</div>` `` →
  `` `<div>${esc(it.name)} ${t('closet.items')}</div>` ``. Mechanical,
  function-by-function.
- Toasts: `showToast('פריט פורסם!')` → `showToast(t('toasts.item_published'))`.
- Hebrew-specific date/number formatting (`toLocaleDateString('he-IL', ...)`,
  the `₪` currency symbol, `DAYS[]` array) needs locale-aware variants too —
  these are not plain strings and need their own small helper
  (`fmtCurrency(n)`, `fmtDate(d)`) keyed off `LOCALE`, not just a `t()` swap.
  Found at minimum: line 2338 (`he-IL` day formatting), `₪` literal currency
  symbol appears in `renderHome`, `renderRewards`, `renderMySells`, and the
  marketplace sell flow — flagging this now so whoever executes doesn't
  treat it as "just another string."

## 3. Screen-by-screen execution order

**Chosen ordering principle: highest-traffic + highest-regression-blast-radius
first, while they have the most reviewer attention and freshest context —
NOT lowest-risk-first.** Rationale: this is explicitly called out as a
sprint-sized effort; doing the long tail of low-traffic screens first would
produce a plan that's 80% "done" by file count while the screens that
actually matter to a non-Hebrew user's first impression (Home, Feed, Closet,
Onboarding) are still Hebrew. If the sprint gets cut short, we want the
highest-value screens converted, not the easiest ones. Toasts are pulled out
as their own cross-cutting phase because they're called from every screen
and doing them screen-by-screen means re-touching the same `showToast()`
call sites repeatedly.

## 4. Verification approach per screen

Per `.claude/skills/verify-rendering/SKILL.md` and **Iron Rule #9**
(established in the 2026-06-17 postmortem after two rendering bugs reached
production despite passing JS syntax checks): every commit that changes
`static/index.html` rendering must be verified in a real headless browser
via Playwright before merge — syntax validation alone is explicitly
insufficient.

For **each phase above**, the verification step is:

1. Confirm dev server is up (`curl localhost:8000/`, start via
   `source venv312/bin/activate && uvicorn app:app --reload` if not).
2. Write/reuse a Playwright driver script (per the skill) that:
   - Loads the page with `localStorage.setItem('awear_onboarded','1')` (skip
     onboarding) plus `localStorage.setItem('awear_locale','en')` to force
     the new locale path.
   - Navigates to the specific view under test (`showView('<name>')` via
     `page.evaluate`, matching the phase's screen).
   - Captures `pageerror` events — any non-empty result is a blocker
     (mirrors both postmortem bug classes: TDZ errors from `loadStrings()`
     ordering, and silent layout breakage).
   - Screenshots the screen at viewport 390x844 (existing convention).
3. **Read the screenshot** and confirm:
   - English text actually appears (not the raw key string like
     `nav.home` — that would mean `t()` fell through to its fallback,
     meaning the key is missing from the JSON, a silent failure mode worth
     explicitly checking for, not just "did something render").
   - No Hebrew text remains visible on that screen (the whole point).
   - Layout is intact — text length differences (English often longer than
     Hebrew) can break fixed-width buttons/pills; this is a real risk on
     dense screens like Feed pills and Marketplace tabs.
4. **Repeat once with `awear_locale` set to `he`** to confirm the Hebrew
   path still renders correctly post-refactor (regression check — the goal
   is to add English, not break existing Hebrew users).
5. Only after both locale screenshots are clean does the phase's diff get
   committed.

This means each phase produces **two screenshots minimum** (en + he) plus a
clean `pageerror` log, not just one happy-path check — given that Phase 0's
entire risk is "did the loading mechanism introduce a load-order bug
exactly like the postmortem's RW_KEY TDZ case," the Hebrew-regression check
on Phase 0 specifically is non-negotiable.

1. **Locale selection UX** — is there a settings screen for the user to pick
   a language, or is it auto-detected from `navigator.language` only? No
   such UI currently exists; this plan assumes auto-detect + a

   Should non-Hebrew locales see `$`, or is this app intentionally
   Israel-currency-only even when global-first on language? This affects
   scope of Phase 2 and 6.

   Varan before treating it as permanently out of scope.

## 6. Summary of effort

- 70 functions + static markup containing Hebrew, ~8,850 Hebrew characters
  total across the file, organized into 11 phases + 1 explicit exclusion.
- Existing `static/i18n/en.json`/`he.json` scaffold is reused and extended,
  not replaced — covers ~10-12% of needed keys today.
- New runtime mechanism: one `t(key)` helper + `loadStrings()` fetch + a
  `data-i18n` attribute pass for static markup + dynamic `lang`/`dir`. No
  new library, no build step, consistent with the existing single-file
  architecture.
- Every phase requires two-locale Playwright verification (en + he) per
  Iron Rule #9 before merge — this is non-negotiable given the team's own
  postmortem history of rendering bugs passing syntax checks but breaking
  in-browser.
