## 5. איך אני נכנסת ל-Loop באופן יזום

הבעיה הספציפית: אין mechanism שמגלה שקוד חדש נכנס ל-main ללא token reference. PR review הוא ה-mechanism — אבל אני לא בתהליך ה-PR review בפועל. גבאנה נמצאת שם. אני לא.

מה שאני מגדירה קדימה:

**א. Token Audit שבועי — לא תלוי ב-dispatch:**
כל שבוע, לפי יום קבוע, אני רצה audit עצמי על `static/index.html`:
- כמה hardcoded hex קיימים שאינם בטוקנים?
- כמה שימושים ב-`var(--t-*)`, `var(--sp-*)`, `var(--r-*)`?
- מגמה — האם המספרים יורדים או עולים?

זה לא תלוי ב-dispatch. זה תלוי בי להגדיר אותו כ-standing commitment.

**ב. Token Reference בכל PR של design/CSS:**
כל PR שמכיל CSS — comment אוטומטי ממני עם בדיקה ספציפית: האם נוסף hardcoded color? האם נוסף `font-size` ללא `var(--t-*)`? אם כן — שם ה-token שהיה צריך להיות בשימוש, לא "לא בסדר".

בפועל: עד שאין לי tool של PR webhooks, אני מבקשת מג'ף להוסיף אותי ל-checklist בכל PR שנוגע ב-CSS לפני merge.

**ג. Style Dictionary Pipeline — מה שבאמת צריך לקרות:**
לא להמשיך לשמור `tokens.css` עם כותרת "generated" כשהוא ידני. שתי אפשרויות:
1. לבנות את ה-generator — `style-dictionary build` script בסיסי, `package.json` command, CI step.
2. או — למחוק את הכותרת המטעה ולתעד במפורש שהקובץ מתוחזק ידנית עד שה-pipeline נבנה.

האפשרות השנייה היא לפחות כנה. האפשרות הראשונה היא מה שאמורה להיות שם.

## 6. מה אני מייעלת ל-Cycle הבא

קובץ `scripts/build-tokens.js` שמקרא את `awear-tokens.json` ומייצר `static/tokens.css` ו-`mobile/src/theme/tokens.js`. אין תלות ב-npm package חיצוני בשלב ראשון — script שכותב CSS variables ו-JS constants ידנית מה-JSON. מספיק כדי שה-"DO NOT edit manually" יהיה אמיתי.

**עדיפות 2 — Migration batch ראשון:**
Typography tokens. הם הכי מוגדרים, הכי עקביים, והכי לא בשימוש. להחליף 402 hardcoded `font-size` ב-`var(--t-*)` — לא בבת אחת. בatches של מסך אחד בכל cycle, לפי traffic: home, feed, closet, marketplace.

**עדיפות 3 — אפס `#2a2040`/`#1a1030` בקוד:**
שני צבעים שלא קיימים בשום token. לא variant, לא alias — לא קיימים. כל שימוש בהם הוא hallucinated color שנכנס ללא review. הם צריכים להיות `var(--surface)` או `var(--card)` לפי ההיררכיה הקיימת. זו migration קונקרטית, ניתנת למדידה, ולא דורשת אישור מארק — זו תרגום למה שכבר הוחלט.

**סדר עבודה:**
לא לנסות לפתור הכל בbatch אחד. לא לשבור קוד שעובד. כל שינוי עובר Playwright regression לפני merge. migration בatches, לא flood.
