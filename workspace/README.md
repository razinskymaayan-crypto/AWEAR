# Workspace — איך עבודה זורמת בפועל (לא רק בתיאוריה)

נוסף 18.06.2026 לפי הנחיית כרמל. זה לא תיעוד עקרוני — זה התיקיה שבה עבודה אמיתית קורית ונראית.

## המבנה

```
workspace/
├── design/    ← מארק, דולצ'ה, גבאנה, נטה כותבים כאן בלבד
├── mobile/    ← וראן, דנה, רועי כותבים כאן בלבד
├── backend/   ← סטיב, סאם, אורן כותבים כאן בלבד
├── product/   ← איילון, שירה כותבים כאן בלבד
└── board/     ← רק ראשי-צוות מעבירים לכאן (לא ICs ישירות)
    ├── design/
    ├── mobile/
    ├── backend/
    └── product/
```

## הכלל

**כל סוכן: עריכה רק בתיקיית הצוות שלו. קריאה — בכל התיקיות, בלי הגבלה.**
זה לא רק נורמה — זה נאכף בחלקו על ידי tools מוגבלים בכל subagent רשום (`.claude/agents/*.md`), ובחלקו ע"י ביקורת ג'ף לפני promotion ל-board (בדיוק כמו אכיפת worktree isolation — לא הכל ניתן לאכיפה טכנית מוחלטת, אז התהליך הוא קו ההגנה השני).

## תהליך עבודה (4 שלבים)

1. **כתיבה** — סוכן IC כותב proposal בתיקיית הצוות שלו: `workspace/<team>/<task-name>/README.md` (מה, למה, diff/patch, הוכחת אימות) + הקבצים הרלוונטיים.
2. **משיכה וביקורת ע"י ראש הצוות** — ראש הצוות (מארק/וראן/סטיב/איילון) מושך את ה-proposal מהתיקייה, עובר עליו ביקורתית. אם צריך שיפור — חוזר ל-IC. Peer review בין ICs קורה כאן, כתועד ב-`org.md`.
3. **קידום ל-board** — אחרי שראש הצוות מאשר, הוא (לא ה-IC) מעביר/מעתיק את ה-proposal ל-`workspace/board/<team>/`.
4. **החלטה סופית** — ג'ף קורא את מה שב-board, מתייעץ עם שאר ראשי הצוותים (קונפליקט/חפיפה בין צוותים נראה כאן), ומחליט אם וכיצד לשלב לאפליקציה האמיתית (`static/index.html`, `app.py` וכו'). זה השלב היחיד שבו קוד נכנס בפועל למוצר החי.

## למה תיקיות ולא רק worktrees

worktrees נשארים המנגנון לבידוד עריכה (כל סוכן עדיין עובד ב-worktree משלו כשהוא כותב קוד). ה-workspace הוא שכבה **נוספת**, נראית-לעין, של staging/review — כדי שכרמל ומעיין יוכלו לדפדף ולראות עבודה בתהליך בלי לחפש ב-git history, וכדי שלכל סוכן תהיה גישת קריאה לעבודה של כולם (העברת ידע, למידה הדדית).
